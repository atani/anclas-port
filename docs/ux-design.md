# Anclas App UX 設計

`design.md` のデータ・アーキテクチャを前提に、**選手情報と今後の試合をキレイに整理して見やすく**を中心に UX を設計する。

設計原則は Apple HIG（Human Interface Guidelines）に従う。ターゲットはアンクラスのサポーター（年齢層は幅広い＝大きめの文字・タップ領域・低い学習コスト）。

---

## 1. 情報設計（IA）

### タブ構成（5タブ・下部 TabBar）

```
┌─────┬─────┬─────┬─────┬─────┐
│ ホーム │ 日程 │ 順位 │ 選手 │ もっと │
│  🏠  │ 📅  │ 📊  │ 👤  │ ⋯   │
└─────┴─────┴─────┴─────┴─────┘
```

| タブ | 役割 | なぜこの位置か |
|---|---|---|
| **ホーム** | 次の試合 + 直近結果 + 最新ニュース1件。開けば全部わかる入口 | 起動直後の3秒価値（idea-killer Q6） |
| **日程** | 今後の試合 + 過去の結果を時系列で。節ごとに整理 | 「次いつ？」「結果は？」の最頻ニーズ |
| **順位** | Qリーグ1部 順位表。アンクラスをハイライト | 順位は1画面で完結 |
| **選手** | 背番号順グリッド → 選手詳細 | ファンが一番「眺めて楽しむ」場所 |
| **もっと** | ニュース一覧 / ポッドキャスト / クラブ情報 / 設定 | 低頻度を集約してメインを軽く保つ |

**設計判断**: ニュースとポッドキャストを独立タブにせず「もっと」に集約する。理由＝メインの4タブ（ホーム/日程/順位/選手）を「試合を追う」体験に集中させ、情報過多を避ける（S-07 の引き算思想を継承）。

## 2. ホーム画面（起動直後の体験）

```
┌──────────────────────────────┐
│  福岡J・アンクラス        🔔     │ ← ヘッダー（クラブカラー）
├──────────────────────────────┤
│  NEXT MATCH                   │
│  ┌────────────────────────┐  │
│  │   Qリーグ 第11節          │  │
│  │                          │  │
│  │  🔵 アンクラス             │  │ ← ホーム/アウェイをアイコン色で
│  │      VS                  │  │
│  │  ⚪ 水俣ユニオンFCW        │  │
│  │                          │  │
│  │   6/27 (金) 12:00 KO     │  │
│  │   📍 会場名               │  │
│  │                          │  │
│  │   ⏱ あと 2日             │  │ ← カウントダウン
│  │  [カレンダー追加] [地図]   │  │
│  └────────────────────────┘  │
│                               │
│  LATEST RESULT                │
│  ┌────────────────────────┐  │
│  │ アンクラス 1 - 0 FC琉球   │  │
│  │ 6/7 Qリーグ第9節  得点:◯◯ │  │
│  └────────────────────────┘  │
│                               │
│  NEXT EVENT                   │
│  ┌────────────────────────┐  │
│  │ 🎙 新着ポッドキャスト        │  │ ← 試合以外の「次にある何か」
│  │ 「ロッカールーム」#12 →    │  │
│  └────────────────────────┘  │
│                               │
│  NEWS                         │
│  ・[最新ニュースタイトル] →     │
└──────────────────────────────┘
```

**ポイント**
- **最重要の3情報（次の試合・直近結果・次回イベント）を最上部に大きく置く**。スクロールせずに「次いつ、前回どうだった、次に何がある」が分かる状態を最優先にする
- 次の試合を**カード1枚で完結**。日時・対戦・会場・カウントダウン・アクション（カレンダー/地図）まで指1本で
- カウントダウンが「生きてる感」を出す（試合がない時期は「次節調整中」表示でオフシーズンも破綻させない）
- 直近結果は1枚だけ。詳細は日程タブへ
- **次回イベント枠**は、試合以外の予定（ポッドキャスト新着・ファンイベント・チケット発売）をホームに集約する。Phase 1 では試合リマインドのみ、Phase 2 でポッドキャスト新着を流す
- データは中継側（GitHub Actions）が `matches.json` に `anclas.nextMatch` / `anclas.latestResult` を派生計算して埋める。アプリは最上部にそのまま大きく描画するだけでよい

## 3. 日程タブ（今後の試合をキレイに整理）

ユーザー要望の核その1。q-league の全試合データを**節ごとのタイムライン**に整理する。

```
┌──────────────────────────────┐
│  日程・結果      [今後｜全試合]   │ ← セグメント切替
├──────────────────────────────┤
│  ●  第11節                    │
│  ┌────────────────────────┐  │
│  │ 6/27(金) 12:00          │  │
│  │ 🔵アンクラス vs 水俣⚪    │  │ ← 未消化は「vs」
│  │ 📍会場名         [詳細]  │  │
│  └────────────────────────┘  │
│                               │
│  ●  第10節  ✓終了             │
│  ┌────────────────────────┐  │
│  │ 6/14(土)                │  │
│  │ ⚪ヴィアマテラス 0-1 🔵   │  │ ← 終了はスコア、勝敗で色
│  │ 🟢 WIN                   │  │ ← 勝=緑/分=灰/負=赤の帯
│  └────────────────────────┘  │
│                               │
│  ●  第9節   ✓終了             │
│  ┌────────────────────────┐  │
│  │ 🔵アンクラス 1-0 FC琉球⚪ │  │
│  │ 🟢 WIN  得点: ◯◯         │  │
│  └────────────────────────┘  │
└──────────────────────────────┘
```

**整理のルール**
- **時系列の縦タイムライン**。左に節マーカー（●）、未来は上 or 下のどちらかに統一（次の試合が最上部）
- **ホーム/アウェイを色で即判別**（🔵=アンクラス側、相手は⚪）。アンクラスを常に左に固定すると、ホーム/アウェイが分かりにくい → **実際のホーム/アウェイ通りに並べ、自チームだけ青で塗る**
- **勝敗を色帯で一目化**（🟢WIN / ⬜DRAW / 🟥LOSE）。スクロールするだけでシーズンの調子が見える
- セグメントで「今後だけ / 全試合」を切替
- 未消化＝「vs」、終了＝スコア。q-league の `【vs】`/`【2-2】` がそのまま対応

**カレンダー連携**: 各試合カードから端末カレンダーに追加（EventKit）。アウェイ遠征の予定管理にも効く。

## 4. 選手タブ（選手情報をキレイに整理）

ユーザー要望の核その2。リッチな選手データを「眺めて楽しい」形に。

│  ┌──────┐  ┌──────┐         │

│  │  #3   │  │  #5   │         │ ← 背番号を大きく

│  └──────┘  └──────┘         │
│  ┌──────┐  ┌──────┐         │
│  │ [写真] │  │ [写真] │         │
│  │  #6   │  │  #7   │         │
│  │ 森和奏 │  │  …    │         │

**整理のルール**
- **ヒーロー写真 + 背番号 + 名前**で「推し」を主役に
- 情報を3ブロックに分類: **PROFILE**（公式情報＝生年月日/出身/身長/サッカー歴）/ **経歴**（クラブ遍歴）/ **PERSONAL**（MBTI・趣味・好きな食べ物＝ファンが楽しむ親近感情報）
- **PERSONAL はチップ表示**でカジュアルに。固いテーブルにしない（ファンエンゲージメントの核）
- **プレーの特長は引用風**に大きく見せる（選手の魅力を1行で）
- 最下部に**選手ブログへの導線**（名鑑×ブログを連結）

## 6. ビジュアルデザイン方針

| 要素 | 方針 |
|---|---|
| **カラー** | アンクラスのクラブカラーを基調（公式サイト・ユニフォームから採取）。採取候補は anclas.jp 頻出のネイビー系 `#273349`（確定は申請前にロゴ・ユニフォームから）。アクセントは勝敗色（緑/灰/赤） |
| **タイポ** | SF Pro（システム）。数字（スコア・背番号・カウントダウン）は太く大きく |
| **写真** | 選手写真・エンブレムを主役に。余白を活かす |
| **ダークモード** | 対応（HIG 標準）。ナイトゲーム観戦でも見やすく |
| **Dynamic Type** | 対応。サポーターの年齢層が広いので文字拡大に追従 |
| **アイコン** | SF Symbols 中心。独自アイコンは最小限 |

## 7. UX 原則（見やすさのための判断基準）

1. **3秒ルール**: どのタブも、開いて3秒で目的の情報に到達できる。特にホームは**次の試合・直近結果・次回イベントを最上部に大きく**置き、スクロール不要で要点が伝わるようにする
2. **1カード1情報**: 試合も選手も「カード」単位。情報を詰め込まない
3. **色で状態を伝える**: 勝敗・ホーム/アウェイ・自チームを色で即判別（文字を読ませない）
4. **数字を主役に**: スコア・背番号・カウントダウン・勝点は大きく
5. **ログイン不要・設定不要**: 開いたら使える。アカウントを作らせない
6. **オフライン耐性**: 最後に取得した JSON をキャッシュ表示（圏外の会場でも見られる）
7. **引き算**: ニュース過多にしない。試合と選手を追う体験に集中（FotMob/OneFootball との差別化）

---

## 8. アクセシビリティ

- VoiceOver 対応（スコアは「アンクラス1対0、勝ち」と読み上げ）
- タップ領域は最低 44×44pt（HIG）
- 色だけに依存しない（勝敗は色帯 + WIN/DRAW/LOSE テキスト併記）
- コントラスト比 WCAG AA 以上

## 9. 画面遷移まとめ

```
[ホーム] ──→ 試合詳細 / ニュース詳細
[日程]  ──→ 試合詳細（スコア・得点者・会場地図・カレンダー追加）
[順位]  ──→ （展開で得失点差）
[選手]  ──→ 選手詳細 ──→ 選手ブログ
[もっと] ─→ ニュース一覧 / ポッドキャスト(→Spotify) / クラブ情報 / 設定(通知)
```
